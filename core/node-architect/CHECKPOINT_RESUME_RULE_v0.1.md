# Checkpoint Resume Rule v0.1

## Status

- Rule ID: `CHECKPOINT_RESUME_RULE`
- Version: `0.1`
- Scope: `REVAMP_UPGRADE_GWC`
- Introduced by: `REVAMP-GWC-003`
- Lifecycle: additive node-architect runtime rule

## Purpose

This rule defines the minimum durable checkpoint and resume token contract for
GWC-governed coding work.

It exists to prevent agents from losing gate context, repeating approval
requests, or resuming from external systems that are only audit projections.

## Core principle

```text
Resume authority = canonical checkpoint + current repo evidence + exact next action.
External systems = hints only.
```

A resume token may point to Jira, TC, DS MCP, comments, labels, dashboards, or
Notion context, but those references never replace the canonical checkpoint.

## Required checkpoint content

Every durable checkpoint MUST record:

| Field | Meaning |
|---|---|
| `checkpoint_id` | Stable checkpoint identifier |
| `task.id` | GWC task/work item |
| `repository.full_name` | Target repository |
| `repository.base_branch` | Protected base branch |
| `repository.base_sha` | Protected base SHA used for scope |
| `gate.current` | Current gate |
| `gate.status` | Current gate status |
| `scope.files_read` | Files inspected |
| `scope.files_write` | Files authorized for write |
| `git_delivery` | Branch, PR, head SHA, CI evidence when available |
| `validation` | Performed/skipped validation with evidence |
| `next_action` | Exact next intended step |
| `audit_projection` | Optional non-authoritative links/hints |

## Resume token requirements

A resume token MUST:

1. bind to exactly one checkpoint by `checkpoint_id`;
2. include `issued_at_utc` and `expires_at_utc`;
3. include the next gate/action that the agent may prepare;
4. include whether human approval is still required;
5. never contain production secrets, credentials, or private data;
6. never mark external projection as canonical gate authority.

## Gate behavior

| Gate | Resume behavior |
|---|---|
| G0/G1 | May resume from checkpoint only as preparation unless formal artifacts exist |
| G2 | Requires exact active approval before repository write |
| G3 | Requires exact PR/head/CI evidence before G4 request |
| G4 | Requires exact human approval for current PR head SHA |
| G5 | Read-only status verification unless deploy/release is explicitly approved |
| G6 | Not applicable unless production data/config/secrets are explicitly in scope |

## Hard invalid states

A checkpoint or resume token is invalid when:

- `external_audit_projection` is treated as source of truth;
- a resume token references a missing checkpoint;
- the protected base SHA is missing;
- the write scope is missing for a repository write task;
- a merge/deploy/production action is next without its own gate approval;
- the approval command is hand-written by a human instead of generated by the agent.

## Compatibility

This rule is additive. It does not remove the existing GWC gate lifecycle, Kiro
checkpoint expectations, FastLane boundaries, or PR/CI evidence requirements.
